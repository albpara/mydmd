# AI Assistant Guidance - RetroPixelLED Repository

This file provides guidance for AI assistants working with this ESP32 LED matrix + WiFi portal project.

## Project Summary
**RetroPixelLED** is an ESP32 embedded system that:
- Controls 2 HUB75 RGB LED panels (128x32 combined resolution)
- Provides a WiFi captive portal for easy network configuration
- Displays animated scrolling text with rainbow HSV colors and optional digital clock
- Stores WiFi credentials persistently using ESP32 Preferences

## File Organization & Purpose

### Core Files
- **`src/main.cpp`** - Application entry point, setup(), loop(), mode switching logic
- **`src/config.h`** - Hardware pin definitions (HUB75 color/row/control pins), constants
- **`src/wifi_manager.h`** - WiFi initialization, AP/STA mode handling, credential loading, NTP time sync
- **`src/web_server.h`** - Async web routes: "/" (portal), "/api/scan-wifi", "/api/connect-wifi"
- **`src/display_manager.h`** - LED rendering functions: scrollText, clock display, HSV color conversion
- **`src/portal_html.h`** - Embedded HTML/CSS/JS for the captive portal UI (retro styling)

### Configuration
- **`platformio.ini`** - Build config, libraries, serial speed (115200), upload speed (921600)

### Documentation
- **`README.md`** - Spanish overview of hardware pins and basic usage
- **`QUICKSTART.md`** - 5-step quick start guide with common troubleshooting
- **`TECHNICAL_SUMMARY.md`** - Architecture details and implementation notes
- **`CAPTIVE_PORTAL_GUIDE.md`** - Portal mechanics and configuration options

## Hardware Configuration
HUB75 is a standard RGB LED panel interface. This project uses:
- **2 chainable 64x32 panels** wired as single 128x32 display
- **DMA-based I2S driver** (mrcodetastic library) for non-blocking LED updates
- **13 GPIO pins** for color (6), row select (4), and control (3)

Pin mapping: See `src/config.h` lines with `#define *_PIN`

## Workflow & Modes

### Startup Logic
1. Load saved WiFi credentials from Preferences storage
2. If credentials exist → Try STA connection (non-blocking, 10s timeout in loop)
3. If no credentials or connection fails → Start AP mode @ 192.168.4.1
4. Always start web server on port 80
5. Start DNS spoofing only if in AP mode (captive portal redirect)

### Display Modes
- **Mode 0**: Digital clock (requires WiFi + NTP sync)
- **Mode 1**: Scrolling text with rainbow animation
- **Mode switching**: Auto-cycles every 10 seconds if both enabled (configurable)

### Web Portal Interaction
User flow: Connect to "RetroPixelLED" SSID → auto-redirect to 192.168.4.1 → select network → enter password → credentials saved → device reconnects → portal stops

## Common Modification Patterns

### Changing AP SSID or Password
- Location: `src/main.cpp` line ~16
- Variables: `WIFI_SSID`, `WIFI_PASSWORD`

### Changing Default Scrolling Text
- Config default: `src/config.h` `DEFAULT_SCROLL_TEXT`
- Runtime modification: Web API POST to `/api/connect-wifi` or Preferences in code

### Adjusting LED Brightness
- Location: `src/config.h` `BRIGHTNESS` (value 10-255)
- Note: USB power limited to ~10%, mains power allows higher values

### Changing GPIO Pins
- Edit all `*_PIN` defines in `src/config.h`
- Rebuild and upload

### Modifying Portal UI
- Location: `src/portal_html.h` in the `getPortalHTML()` string
- Inline HTML/CSS/JavaScript (embedded as C++ string literal)

## Build & Deployment

### Prerequisites
- PlatformIO CLI installed
- USB connection to ESP32

### Commands
```bash
# Compile only
pio run -e esp32dev

# Upload to device
pio run -t upload -e esp32dev

# Monitor serial output
pio device monitor -e esp32dev -b 115200

# Combined: compile and upload
pio run -e esp32dev; pio run -t upload -e esp32dev
```

## Debugging Tips
- Serial output (115200 baud) logs all major operations with `[PREFIX]` tags
- Common prefixes: `[BOOT]`, `[WIFI]`, `[WEB]`, `[API]`, `[DNS]`, `[NTP]`, `[LOOP]`, `[MQTT]`
- Check serial monitor to diagnose WiFi connection or portal issues
- MQTT diagnostics: Look for `[MQTT] ✓ Discovery message published successfully` in logs

## MQTT Configuration & Home Assistant Integration
The device includes built-in Home Assistant MQTT auto-discovery. Configure via the captive portal form:

### Required Fields:
- **Broker Address** - IP or hostname of MQTT broker
- **Broker Port** - Usually 1883 (plain) or 8883 (TLS)
- **Client Name** - Custom MQTT client ID (default: "RetroPixelLED")

### Optional Fields:
- **Username/Password** - If broker requires authentication
- **Topic Prefix** - Prefix for all MQTT topics (default: "retropixel")

### Topics:
- **State**: `{prefix}/display/state` - Reports "ON" or "OFF"
- **Command**: `{prefix}/display/power/set` - Accept "ON"/"OFF" commands
- **Discovery**: `homeassistant/light/{device_id}/display/config` - Auto-discovery message

### Troubleshooting MQTT:
If device doesn't appear in Home Assistant:
1. Check serial logs for `[MQTT] Connected!` message
2. Verify `[MQTT] ✓ Discovery message published successfully`
3. In HA: Settings → Devices & Services → MQTT → Check "Enable discovery"
4. Check MQTT broker logs to see published messages
5. Verify topic structure matches HA expectations (should auto-configure)

## Notes for Future Modifications
- The header-based modular design allows easy feature additions without file restructuring
- Async web server is non-blocking (important for smooth LED animation during WiFi operations)
- HSV→RGB565 conversion in `display_manager.h` enables smooth rainbow color transitions
- Preferences storage is limited (~4KB per namespace); keep credential strings short
